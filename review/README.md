# Reviews

This directory collects occasional **AI-driven analyses of the Guardrails codebase**. Each analysis
is a point-in-time document, generated by an AI coding agent, that examines the project from a
particular angle — how it compares to similar projects, the state of its security posture, and so
on. The intent is to keep this kind of assessment **honest and public**: the findings, their
sources, and the exact commit they correspond to are recorded alongside the analysis, so a reader
can judge the work for themselves and see how the project's standing evolves over time.

These documents are **development artifacts of the Guardrails repository itself**, not part of the
distributed template. They live only at the repository root and are never rendered into generated
projects or updated by `copier update`.

## Organization

```
review/
  <category>/
    <YYYY-MM-DD>/
      README.md      # the analysis, rendered as the folder's landing page
```

Each analysis lives in a dated directory named `README.md` so that it renders directly when the
folder is browsed. Categories group analyses by kind:

- **`comparisons/`** — how Guardrails compares to other projects and techniques in its space.
- **`security/`** — *(planned)* security-focused assessments of the project.

Further categories can be added as the same pattern (`review/<category>/<date>/README.md`).

## Provenance and honesty

Every analysis records, near its top:

- the **goal** of the analysis,
- the **date** it was produced,
- the **commit** it corresponds to (and the working-tree state at that time), and
- the **tool** that generated it.

Because these are AI-generated and cite third-party, fast-moving material, they should be read as
**point-in-time assessments**, not maintained references: sources may move, adoption figures drift,
and individual claims can be wrong even after verification. Treat the recorded commit and date as the
scope of each document.
